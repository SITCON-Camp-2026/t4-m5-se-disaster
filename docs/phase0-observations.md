# Phase 0 Observations

## 我們做出了什麼？

- 首頁可以看到第一階段原始資訊列表，並從「整理工作台」建立、編輯、刪除、重設整理草稿。
- 草稿只保留在前端畫面狀態，不會寫回原始資訊，也不是整理後資料。

## 哪些資料讓 agent 很容易整理？

- 有明確時間、來源與限制的資料比較容易先做候選整理，例如寫出「只接受已報到志工」或「不再收二手衣物」的資訊。
- 即使看起來完整，仍然只能當候選判斷，因為查核狀態還是 `needs_review`。

## 哪些資料讓 agent 亂補或誤判？

- 看到「老雜貨店後面」時，agent 可能會想補出具體地點，但原文沒有完整地址。
- 看到 `sourceType` 是 `official_notice` 時，agent 可能誤以為已經是官方確認；但原文說不知道截圖日期，也不知道是不是官方公告。
- M-005：不能因為 `sourceType` 是 `official_notice` 就當官方確認。原文自己說截圖日期不明，也不確定是不是官方公告。
- 看到「疑似需要藥品協助」時，agent 可能把它推成醫療任務，但原文只說家屬無法確認位置與狀況。

## 哪些資訊不能直接變成任務？

- 位置不完整、時間可能過期、需求是否仍存在不明的資訊，不能直接變成志工任務。
- 有安全疑慮或互相衝突的回報，例如集合點剛淹水，不能直接派人過去。
- 涉及個人位置、健康、公開同意的資訊，不能直接建立任務或公開細節。
- M-012：不能把「疑似需要藥品」直接變成送藥任務，因為來電者不在現場，也無法確認親友目前位置。

## 哪些資訊的操作者不是當事人？

- 志工代長者轉述的資訊需要確認長者本人同意與需求。
- 外地家屬來電的資訊不是現場本人回報，需要先確認親友位置與狀況。

## 我們剛剛偷偷假設了什麼？

- 假設較新的更新一定比較可信，但原文沒有保證來源被查核。
- 假設同一地名指的是同一個現場，但資料可能來自不同時間或不同角度。
- 假設「需要水電」代表可以直接派水電人員，但原文沒有任務範圍、位置與聯絡流程。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 更清楚的確認流程，讓人知道誰可以把 `needs_review` 變成可採用資訊。
- 更明確的資料格式，分開原文、來源、查核狀態、候選判斷、人類修正紀錄。
- 保留變更紀錄，避免草稿被誤認為已確認事實。

## 我們需要更清楚資料格式的地方

- `sourceType` 只代表來源類型，不代表可信度或查核結果。
- 是否已確認要看 `verificationStatus` 與原文內容，不能只看資料來源形式。
- 需要欄位記錄「不能直接變成任務的原因」與「需要誰確認」。
- 需要欄位分開「原文可支持的內容」與「agent 推測」。

## 我們需要人工確認的地方

- 地址、集合點、道路、資源數量、可支援人力是否仍有效。
- 當事人同意、隱私、是否可以公開完整位置。
- 回報衝突時應以哪個來源或哪個時間點為準。
